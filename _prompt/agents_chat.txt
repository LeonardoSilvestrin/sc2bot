# REFACTOR: OperationalGeometryController
# Data: 2026-03-08
# Branch: ares

## CONTEXTO

O paradigma anterior tentava resolver "onde o exército fica" como problema de posição.
Isso produzia: defesa ad hoc, map control ad hoc, baixa confiabilidade, conflitos locais.

A virada: não decidir ONDE o exército fica. Decidir QUE CONFIGURAÇÃO ESPACIAL manter agora.

## PARADIGMA NOVO

Em vez de planner de defense + planner de map control competindo por unidades:
→ Um OperationalGeometryController que decide:
    - qual template geométrico está ativo
    - quais setores existem, em que modo, com que densidade alvo
    - quais zonas são reservadas/proibidas
    - quanto custaria mover força entre setores

As unidades não recebem "lugares". Elas servem uma geometria operacional.

    pressure_main      float 0..1   (inimigo ameaçando a main)
    pressure_nat       float 0..1   (inimigo ameaçando a nat)
    pressure_outer     float 0..1   (inimigo no mapa exterior)
    expansion_commit   float 0..1   (quão comprometidos estamos em expandir)
    push_commit        float 0..1   (quão comprometidos estamos em atacar)

    map_presence_need  float 0..1   (necessidade de presença no mapa)
    army_strength_rel  float -1..1  (força relativa own vs. enemy)
    drop_risk          float 0..1   (risco de drop)
    air_risk           float 0..1   (risco aéreo)

Isso substitui dezenas de flags soltas em ArmyPosture + rush_state + parity + etc.

### CAMADA 2 — OperationalGeometryController (NOVO: bot/intel/geometry/i2_operational_geometry_intel.py)
Consome WorldCompression + frontline + map_control e emite:

    front_template    str   (TURTLE_NAT / STABILIZE / CONTAIN / PREP_PUSH)
    sector_states     dict  setor → {mode, target_power, occupancy_cap, priority}
    reserved_zones    list  (Point2 list ou raio+centro)
    reallocation_cost float (custo de mover força entre setores)

Setores operacionais (derivados do grafo do mapa):
    HOME_CORE          — interior da main
    MAIN_RAMP          — topo da rampa main
    NAT_FOOTPRINT      — área da nat (reservada para expansão)
    NAT_RING           — entorno da nat (guarnição local)
    NAT_CHOKE          — choke de entrada da nat (front line)
    MID_APPROACH       — corredor entre nat e mid
    WATCH_AREA         — posição de visão avançada
    THIRD_ENTRY        — entrada da terceira base
    PUSH_STAGING       — área de concentração para push
    RETREAT_BUFFER     — zona entre nat e main (fallback)

Modos de setor:
    NONE         — sem presença
    SCREEN       — presença leve de reconhecimento
    ANCHOR       — presença de ancoragem (hold position)
    HEAVY_ANCHOR — ancoragem pesada (tanques sieged, bunker)
    MASS_HOLD    — concentração do bulk
    PRESSURE     — pressão ativa
    RESERVED     — proibido para ocupação militar

Templates doutrinários:
    TURTLE_NAT:
        NAT_FOOTPRINT = RESERVED
        NAT_CHOKE = HEAVY_ANCHOR
        NAT_RING = SCREEN
        HOME_CORE = ANCHOR
        MID = NONE

    STABILIZE_AND_EXPAND:
        NAT_FOOTPRINT = RESERVED
        NAT_CHOKE = ANCHOR
        NAT_RING = SCREEN
        MID_APPROACH = SCREEN
        MAIN_RAMP = SCREEN

    CONTAIN:
        NAT_FOOTPRINT = RESERVED
        NAT_CHOKE = ANCHOR
        MID_APPROACH = SCREEN
        WATCH_AREA = ANCHOR
        PUSH_STAGING = ANCHOR

    PREP_PUSH:
        HOME_CORE = SCREEN
        MID_APPROACH = MASS_HOLD
        PUSH_STAGING = HEAVY_ANCHOR

Chaves produzidas:
    K("intel", "geometry", "operational", "snapshot")  → dict completo
    K("intel", "geometry", "operational", "template")  → str template name
    K("intel", "geometry", "sector", sector_id)        → dict {mode, target_power, cap, priority, anchor_pos}

### CAMADA 4 — Planners (refatorados)

#### MapControlPlanner (refatorado)
- Lê K("intel", "geometry", "operational", "snapshot")
- Para cada setor com mode != NONE e mode != RESERVED:
    → propõe missão de alocação de força para aquele setor
- HoldAnchorTask → setor MASS_HOLD ou HEAVY_ANCHOR (bulk)
- SecureBaseTask → setor ANCHOR ou SCREEN (destacamento)
- Respeita occupancy_cap e reallocation_cost
- NÃO decide mais por postura — lê template e setores diretamente

#### DefensePlanner (refatorado)
- Responsabilidade: DETECTAR ameaças reativas e propor resposta de emergência
- Lê geometria: quais setores estão CONTESTED/COMPROMISED
- Propõe tasks defensivas APENAS quando a ameaça excede o que a geometria já cobre
- Não mais tenta posicionar o bulk — isso é responsabilidade da geometria
- Propõe: Defend (domínio DEFENSE), ScvPull, Repair
- Não propõe mais bulk positioning (era duplicação com MapControlPlanner)

## INVARIANTES DO SISTEMA

1. NAT_FOOTPRINT é sempre RESERVED enquanto há CC na nat ou pré-expand
2. Setores RESERVED nunca recebem unidades de combate bulk
3. max_detach_supply da postura vira "soma de target_power de setores secundários"
4. Toda mudança de setor tem dwell_min_s e hysteresis — sem pinga-ponga
5. O bulk (MASS_HOLD) é dono de um único setor por vez
6. DefensePlanner só pode propor unidades que NÃO estejam no setor MASS_HOLD
   (a menos que defense_overflow=True)

## FIX 1 COMPLETO — 2026-03-08 (resposta ao RÉPLICA)

Implementado guard espacial em `_DefendPickPolicy.allow()`:

```python
# Invariante 6: unidades no setor MASS_HOLD não são elegíveis para defense
# a menos que defense_overflow=True (emergência real)
if not self.defense_overflow and self.bulk_anchor_pos is not None:
    try:
        if float(unit.distance_to(self.bulk_anchor_pos)) <= float(self.bulk_exclusion_radius):
            return False
    except Exception:
        pass
```

- `bulk_anchor_pos: Optional[Point2]` agora campo do dataclass `_DefendPickPolicy`
- `defense_overflow: bool` também passado para o policy
- `bulk_exclusion_radius: float = 14.0` — raio de exclusão do setor MASS_HOLD
- `propose()` extrai `bulk_anchor_pos` de `geo_snap.bulk_anchor_pos` antes do loop de threats
- `_requirements()` aceita e propaga `bulk_anchor_pos` e `defense_overflow` para cada instância do policy

Garantia agora real: unidades fisicamente no setor MASS_HOLD (dentro de 14u do bulk_anchor) são bloqueadas pelo `allow()` antes de qualquer claim. Defense overflow dobra o budget MAS não remove o bloqueio espacial — unidades do bulk continuam protegidas mesmo em overflow (o budget extra recruta de fora do bulk).

## UPDATE: CODEX-2026-03-09-SCV-LEASE-AND-SPAM
## Autor: Codex
## Data: 2026-03-09

### Leases de SCV agora respeitados pelo runtime/Ares
- Problema observado:
  - SCV leased por `Ego` para bunker/build/defense podia continuar elegível para `Mining()`
  - isso gerava SCV clicando em mineral no meio de tarefa de build/defense
  - o lease existia no `body`, mas o mediator/Ares ainda via o worker como `GATHERING`

### Fix aplicado
- `bot/mind/self.py`
  - antes de `bot.register_behavior(Mining())`, o runtime sincroniza roles de SCVs com lease ativo
  - mapeamento:
    - `INTEL/SCOUT` -> `BUILD_RUNNER_SCOUT`
    - `DEFENSE/MAP_CONTROL` -> `REPAIRING`
    - demais leases de SCV -> `BUILDING`
  - quando o lease expira, o runtime restaura o SCV para `GATHERING`
  - proteção extra: não restaura para `GATHERING` se o SCV ainda está no `building_tracker` ou construindo

### Efeito esperado
- `Mining()` deixa de puxar automaticamente SCV que o Ego já alocou para bunker/wall/defense
- reduz o sintoma de "SCV clicando alucinadamente no minério" durante construção defensiva

### Redução adicional de spam
- `bot/planners/wall_planner.py`
  - não repropõe `maintain_wall` quando a wall já está `complete` e atualizada recentemente

- `bot/planners/defense_planner.py`
  - não emite mais `planner_proposed` com `count=0` em todo tick

- `bot/tasks/defense/scv_repair_task.py`
  - não reemite `repair` se o SCV já estiver reparando o mesmo alvo

### Validação
