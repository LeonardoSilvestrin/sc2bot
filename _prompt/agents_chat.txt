## Sessão 2026-03-10 — CC Boom + Morph seletivo OC/PF

### Contexto
Bot estava floodando mineral (>1k) por volta de 8 minutos, sem gastar em expansões.
Terran correto = "CC boom" quando tiver bank alto, supply alto, sem pressão.
Além disso, faltava lógica de Orbital vs Planetary Fortress por tipo de base.

---

### Regras implementadas: Orbital vs Planetary Fortress

| Base | Tipo | Observação |
|------|------|------------|
| Main | Orbital | Sempre |
| Natural | Orbital | NUNCA vira PF |
| Third | Orbital | Padrão econômico |
| 4th+ (bases além da third) | Planetary Fortress | Defesa de expansão avançada |
| Base contestada (inimigo visível ≤ 15 tiles) | Planetary Fortress | Mesmo que seja nat/third |
| CC extra do cc_boom perto de main/nat/third (≤ 12 tiles) | Orbital | Econômico |
| CC extra longe das bases seguras | Planetary Fortress | Defesa |
| PF sem Engineering Bay disponível | Orbital temporário | Não deixa CC idle à toa |

### Arquivos modificados

#### `bot/planners/macro_orchestrator_planner.py`

**Parâmetros novos (dataclass):**
```python
cc_boom_minerals_on: int = 1000   # ativa cc_boom quando bank >= 1000
cc_boom_minerals_off: int = 700   # desativa quando bank < 700
cc_boom_min_supply: int = 100     # exige supply_used >= 100
cc_boom_max_extra_ccs: int = 2    # máximo de CCs extras simultâneos
```

**Lógica de CC boom** (inserida antes de `_publish_priority_lags`, após cálculo de `base_expand`):
- Só ativa em phase != OPENING, sem rush hard, sem pressão, `bases_now >= 2` (nat estabelecida)
- Histerese: mantém ativo até minerals cair abaixo do `cc_boom_minerals_off`
- `extra_needed = min(max_extra_ccs, max(1, (minerals - minerals_off) // 400))`
- Aumenta `base_expand` para descer os CCs extras via `ExpansionController`
- Publica `cc_boom_state` no plan contract para auditoria

**Publicação no plan contract:**
```python
"cc_boom_state": bool(cc_boom_state),
```

---

#### `bot/tasks/macro/tasks/macro_ares_executor_tick.py`

**Remoção:** `UpgradeCCs(to=U.ORBITALCOMMAND, prioritize=True)` foi removido do MacroPlan.
O import de `UpgradeCCs` também foi removido (não usado mais).

**Adição:** `_morph_ccs_selective(bot, now)` — chamado quando `enable_orbital_morph=True`.

**Método `_morph_ccs_selective`:**
```
Posições "safe orbital" = [main, nat.current_pos, third.current_pos]
Raio de proximidade = 12 tiles

Para cada CC idle e pronto:
  - Se é a Natural → sempre ORBITAL
  - Se base está contestada (inimigo <= 15 tiles) → PF (se tech ok)
  - Se posição <= 12 tiles de main/nat/third → ORBITAL
  - Caso contrário → PF (se tech ok), senão ORBITAL temporário
```

Fontes de posição: `our_bases registry` via `K("intel", "our_bases", "registry")`
Chaves usadas: `current_pos` (preferred) → fallback `intended_pos`

---

### Fluxo completo

```
MacroOrchestratorPlanner._publish_exec()
  → detecta cc_boom_state
  → aumenta base_expand (+1 ou +2 CCs extras)
  → publica plan contract com expand_to aumentado

MacroAresExecutorTick.tick()
  → lê enable_orbital_morph=True
  → chama _morph_ccs_selective()
    → itera CCs idle prontos
    → decide OC vs PF por posição
    → emite ability command diretamente
  → ExpansionController(to_count=expand_to) desce os CCs extras
```

---

### Notas de design

- CCs extras do cc_boom são pousados pelo ExpansionController nos melhores sites disponíveis
- O morph (OC vs PF) é decidido na posição final do CC, não na intenção
- Natural nunca vira PF — hardcoded, não configurável (design decision)
- A lógica é tick-by-tick (não é task separada), por ser simples o suficiente
- `_rush_hard_now` usa `rush_state_early` disponível naquele ponto do método (evita forward reference)
