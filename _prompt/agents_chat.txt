## Sessão 2026-03-10 — CC Boom + Morph seletivo OC/PF

### Contexto
Bot estava floodando mineral (>1k) por volta de 8 minutos, sem gastar em expansões.
Terran correto = "CC boom" quando tiver bank alto, supply alto, sem pressão.
Além disso, faltava lógica de Orbital vs Planetary Fortress por tipo de base.

---

### Regras implementadas: Orbital vs Planetary Fortress

| Base | Tipo | Observação |
|------|------|------------|
| Main | Orbital | Sempre |
| Natural | Orbital | NUNCA vira PF |
| Third | Orbital | Padrão econômico |
| 4th+ (bases além da third) | Planetary Fortress | Defesa de expansão avançada |
| Base contestada (inimigo visível ≤ 15 tiles) | Planetary Fortress | Mesmo que seja nat/third |
| CC extra do cc_boom perto de main/nat/third (≤ 12 tiles) | Orbital | Econômico |
| CC extra longe das bases seguras | Planetary Fortress | Defesa |
| PF sem Engineering Bay disponível | Orbital temporário | Não deixa CC idle à toa |

---

### Arquivos modificados

#### `bot/planners/macro_orchestrator_planner.py`

**Parâmetros novos (dataclass):**
```python
cc_boom_minerals_on: int = 1000   # ativa cc_boom quando bank >= 1000
cc_boom_minerals_off: int = 700   # desativa quando bank < 700
cc_boom_min_supply: int = 100     # exige supply_used >= 100
cc_boom_max_extra_ccs: int = 2    # máximo de CCs extras simultâneos
```

**Lógica de CC boom** (inserida antes de `_publish_priority_lags`, após cálculo de `base_expand`):
- Só ativa em phase != OPENING, sem rush hard, sem pressão, `bases_now >= 2` (nat estabelecida)
- Histerese: mantém ativo até minerals cair abaixo do `cc_boom_minerals_off`
- `extra_needed = min(max_extra_ccs, max(1, (minerals - minerals_off) // 400))`
- Aumenta `base_expand` para descer os CCs extras via `ExpansionController`
- Publica `cc_boom_state` no plan contract para auditoria

**Publicação no plan contract:**
```python
"cc_boom_state": bool(cc_boom_state),
```

---

#### `bot/tasks/macro/tasks/macro_ares_executor_tick.py`

**Remoção:** `UpgradeCCs(to=U.ORBITALCOMMAND, prioritize=True)` foi removido do MacroPlan.
O import de `UpgradeCCs` também foi removido (não usado mais).

**Adição:** `_morph_ccs_selective(bot, now)` — chamado quando `enable_orbital_morph=True`.

**Método `_morph_ccs_selective`:**
```
Posições "safe orbital" = [main, nat.current_pos, third.current_pos]
Raio de proximidade = 12 tiles

Para cada CC idle e pronto:
  - Se é a Natural → sempre ORBITAL
  - Se base está contestada (inimigo <= 15 tiles) → PF (se tech ok)
  - Se posição <= 12 tiles de main/nat/third → ORBITAL
  - Caso contrário → PF (se tech ok), senão ORBITAL temporário
```

Fontes de posição: `our_bases registry` via `K("intel", "our_bases", "registry")`
Chaves usadas: `current_pos` (preferred) → fallback `intended_pos`

---

### Fluxo completo

```
MacroOrchestratorPlanner._publish_exec()
  → detecta cc_boom_state
  → aumenta base_expand (+1 ou +2 CCs extras)
  → publica plan contract com expand_to aumentado

MacroAresExecutorTick.tick()
  → lê enable_orbital_morph=True
  → chama _morph_ccs_selective()
    → itera CCs idle prontos
    → decide OC vs PF por posição
    → emite ability command diretamente
  → ExpansionController(to_count=expand_to) desce os CCs extras
```

---

### Notas de design

- CCs extras do cc_boom são pousados pelo ExpansionController nos melhores sites disponíveis
- O morph (OC vs PF) é decidido na posição final do CC, não na intenção
- Natural nunca vira PF — hardcoded, não configurável (design decision)
- A lógica é tick-by-tick (não é task separada), por ser simples o suficiente
- `_rush_hard_now` usa `rush_state_early` disponível naquele ponto do método (evita forward reference)

---

### Dívida técnica gerada
- Nenhuma nova dívida crítica
- O `cc_boom_state` poderia futuramente ser lido pelo `AdvantageGameStatusIntel` para modular peso de expansão
- Se o bot tiver 4 bases e flood ainda ativo, o cc_boom vai pedir 5a base — considerar cap por fase

## Sessão 2026-03-11 — Late Game Push: MoveOutTask + Ataque Multi-Frente

### Contexto
Bot ficava parado com 200/200 e banco cheio. Precisava de um move-out late-game
com leapfrog de tanks, bio/mech segurando a frente, e harass paralelo nas bases laterais.

---

### Arquitetura implementada

#### Frente central: `MoveOutTask` (`bot/tasks/map_control/move_out_task.py`)

Task nova que substitui `HoldAnchorTask` (score=97 vs 93) quando o push é ativado.

**3 fases:**
```
CONCENTRATE → ADVANCE (leapfrog) → SIEGE_BASE
```

**Leapfrog de tanks:**
1. Screen (bio/mech) faz `attack_move` para o waypoint atual
2. Tanks (unsieged) correm para `leapfrog_pos` = waypoint + 6 tiles à frente
3. Tanks chegam → siegam
4. Quando todos os tanks siegados (ou timeout de 12s) → screen avança para próximo waypoint
5. Repete com 3 saltos intermediários até a base inimiga

**Commit total:**
- NÃO recua por dano ou urgency moderada
- Só dá `done()` se bulk perdeu >= 60% do supply inicial (colapso de força)
- Na fase SIEGE_BASE: tanks siegam, screen ataca estruturas prioritárias
- Se base limpa → procura próxima estrutura inimiga e atualiza `target_pos`

**Tipos de unidade:**
- `_SCREEN_TYPES`: Marine, Marauder, Hellion, Cyclone, Thor, Widowmine
- `_TANK_TYPES`: SiegeTank, SiegeTankSieged
- Medivacs seguem o centro do screen em todas as fases

---

#### Orquestração: `MapControlPlanner` (modificado)

Novos imports: `MoveOutTask`, `BansheeHarass`, `MedivacDropHarassTask`

**Condições de ativação do push:**
```python
template == PREP_PUSH
AND supply_used >= 160
AND (minerals >= 400 OR vespene >= 200)
AND não há MoveOutTask já rodando
```

**3 proposals paralelas (domínios diferentes — não competem entre si):**

| Proposta | Domínio | Score | Condição |
|----------|---------|-------|----------|
| MoveOutTask | MAP_CONTROL | 97 | sempre que push ativo |
| BansheeHarass (base lateral) | HARASS | 82 | banshee_count >= 1 |
| MedivacDropHarassTask (2a frente) | HARASS | 80 | medivac idle >= 1 + bio idle >= 4 + sem banshee |

**Lógica de alvo:**
- Push target: estrutura inimiga mais distante da main inimiga (= base mais externa conhecida)
- Fallback: expansion mais próxima do inimigo → enemy_start_location
- Banshee target: nat inimiga (diferente do push_target) → enemy_main
- Drop target: 1a expansion inimiga mais próxima da main
- Staging: sector PUSH_STAGING anchor da geometria → fallback ao anchor atual do bulk

---

### Fluxo completo

```
i2_operational_geometry_intel
  → template = PREP_PUSH (quando supply >= 14, push_commit >= 0.4, nat tomada)

MapControlPlanner.propose()
  → push_conditions_met = True
  → propõe MoveOutTask (MAP_CONTROL 97) — bate HoldAnchorTask (93)
  → propõe BansheeHarass (HARASS 82) se tiver banshees
  → propõe MedivacDrop (HARASS 80) se tiver medivac+bio e sem banshee

Ego aceita as 3 proposals (domínios distintos)

MoveOutTask.on_step():
  CONCENTRATE: concentra no PUSH_STAGING (~20s timeout)
  ADVANCE: leapfrog 3x até a base
  SIEGE_BASE: tanks siegam, screen ataca estruturas

BansheeHarass / MedivacDrop: independentes, atacam bases laterais em paralelo
```

---

### Calibração futura (dívida técnica)
- `_COLLAPSE_THRESHOLD = 0.40` — threshold de recuo por colapso (40% supply restante)
- `_LEAPFROG_DELTA = 6.0` — tiles que tanks avançam à frente do screen
- `_LEAPFROG_MAX_WAIT_S = 12.0` — tempo máximo esperando tanks siegarem antes de avançar
- `n_leapfrog_steps = 3` — quantidade de saltos intermediários (configurável)
- Condição de push no planner: `supply >= 160` e `minerals >= 400` — ajustar após testar
- Drop só proposto quando sem banshee — pode mudar para sempre propor se tiver unidades suficientes
- `push_commit` ainda não é escrito por nenhum planner (só lido pelo i2) — o i2 ativa PREP_PUSH por outros sinais
